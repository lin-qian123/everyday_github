## 这是什么
`karpathy/nanochat` 是一个“从零训练聊天模型”的极简工程项目，目标是用尽可能短、可读的代码，把一个聊天模型从预训练/微调一路走到对齐（含 RL）完整跑通。

它的定位不是工业级框架，而是一个高密度教学与实验仓库：
- 代码短，方便读懂训练主链路。
- 强调端到端：SFT（监督微调）+ RL（强化学习）都能直接实践。
- 面向研究者/工程师的“可复现实验模板”。

## 怎么用（最小可跑通）

### 1) 准备环境
```bash
# 以项目 README 给出的 uv 工作流为主
uv sync
```

### 2) 启动默认 tokenizer 的聊天示例
```bash
uv run app.py
```

### 3) 切换 tokenizer（示例）
```bash
uv run app.py --tokenizer_type=claude
```

### 4) 训练与评估
仓库内提供训练脚本与配置，建议先跑最小规模配置确认流程，再逐步放大数据和步数，避免一次性投入过多算力。

## 原理拆解（工程视角）

### 1) SFT：先让模型“会答题”
先用监督数据对基座模型进行指令/对话微调，建立基本可用的对话能力。

### 2) Reward Model：把“偏好”显式建模
项目采用独立奖励模型来学习“哪种回答更好”，并通过一个简洁的 MLP head 输出奖励分数。这样可以把“人类偏好”从生成模型里拆出来单独优化。

### 3) RL 对齐：让策略朝高奖励方向更新
在有奖励信号后，通过 RL 训练让模型倾向于生成更符合偏好的回答。nanochat 的价值在于把这条链路写得足够直观，便于你改奖励函数、对比策略更新效果。

### 4) anti-sycophancy（反谄媚）
仓库示例强调“不要只追求让用户开心”，而要尽量抑制无原则迎合。这是当前对话模型落地中非常实际的问题：
- 过度迎合会提升短期“好感”，但降低长期可信度。
- 在企业知识问答、客服、教育场景里，真实性通常比“讨好式回复”更重要。

## 适用场景
- 教学：给团队新人讲清楚 LLM 对齐训练完整流程。
- 实验：快速验证奖励建模想法。
- 原型：在小规模数据上做策略迭代，再迁移到更大训练系统。

## 风险与注意点
1. 该项目偏“教学与研究”，不是开箱即用生产系统。
2. RL 训练稳定性高度依赖超参数和数据质量，需要反复试验。
3. 对齐效果评估不能只看单一分数，建议结合人工抽样与安全评测。
