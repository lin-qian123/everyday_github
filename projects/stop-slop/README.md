# stop-slop

## 项目定位
`stop-slop` 是一个聚焦“减少 LLM 生成废话/套话（slop）”的提示与输出约束工具型项目，目标是让模型输出更短、更可执行、更工程化。

- GitHub: https://github.com/hardikpandya/stop-slop
- 热度参考：GitHub Trending（2026-05-27 日榜）上榜项目。
- 证据等级：A（榜单）/ C（功能细节仍需进一步复核仓库 README）。

## 用法

1. 克隆仓库并阅读 `README`，优先确认它提供的是提示模板、规则集还是可执行脚本。
2. 将其规则嵌入你的 AI 开发流程（如代码生成、文档生成、Agent 响应规范）。
3. 先在单任务场景 A/B 测试：对比“默认输出”与“约束后输出”的长度、可执行性和返工率。

## 原理

- **约束优先**：通过更严格的输出格式与禁用词/禁用风格，减少“听起来正确但无行动价值”的内容。
- **可验证输出**：鼓励模型给出结构化、可检查、可落地的答案，而非泛化叙述。
- **反馈闭环**：把“过度铺垫、重复表达、无效总结”转化为可量化的质量信号。

## 价值

- 对工程团队而言，可明显降低 AI 输出的“阅读成本”和“二次改写成本”。
- 适合作为团队统一提示规范的一部分，提升多人协作时的输出一致性。
- 在自动化流程中可提高下游工具执行成功率（例如脚本生成、PR 说明生成）。

## 风险边界

- 约束过强会压缩模型的解释能力，可能导致关键信息被省略。
- 不同任务对“简洁”的容忍度不同，统一阈值可能不适配复杂分析任务。
- 当前细节主要来自榜单与公开索引，具体机制需以仓库文档为准。

## 补充建议

- 建议先定义团队内部“slop”判定标准（长度、重复率、行动项密度），再引入此类工具。
- 与代码评审或文档模板联动，形成“生成-检查-回退”的最小闭环。
- 对关键场景保留“详细模式”开关，避免误伤复杂问题解释。
