## 1. 为什么值得跟踪

`everything-claude-code` 的定位不是“单点工具”，而是围绕 Claude Code 的最佳实践与可复用资产集合，适合把 AI 编程能力标准化到团队工作流里。

- 热度信号（最近可验证快照）:
  - GitHub Trending 镜像记录为 `3,724 stars today`（见 2026-03-26 的跟踪报告）
- 传播原因:
  - 开发者关注点从“模型对话”转向“可执行工程流程”；该类仓库通常提供提示词模板、命令范式、项目结构与协作规范，能直接提升落地效率。

## 2. 项目在做什么（核心定位）

从命名与同类仓库形态看，这类项目的核心价值一般在以下四层：

1. 指令与提示模板层
- 把高频开发任务（重构、测试、评审、文档、调试）封装成可复用提示模板。

2. 工作流编排层
- 定义“需求 -> 实现 -> 自测 -> 提交”标准路径，减少每次从零组织上下文的成本。

3. 质量约束层
- 给出代码评审清单、提交规范、风险检查项，降低 AI 代码引入回归的概率。

4. 知识资产层
- 沉淀可迭代的团队实践（例如问题模式库、常见故障修复手册、项目脚手架）。

## 3. 怎么用（建议落地方式）

### 3.1 快速试用

```bash
git clone https://github.com/affaan-m/everything-claude-code.git
cd everything-claude-code
```

先看仓库中的 `README`、`examples`、`templates`、`prompts`（若有）目录，挑 1 个最贴近你当前项目的模板开始。

### 3.2 接入你的项目

1. 选 2-3 个最常用任务模板（如“新增功能”“修复 bug”“代码评审”）。
2. 把模板改成你项目术语（技术栈、目录结构、测试命令、提交规范）。
3. 固化为团队标准入口（如 `docs/ai-playbook.md`）。

### 3.3 验证是否有效

- 对比接入前后:
  - 从需求到首个可运行 PR 的时间
  - 回归 bug 数量
  - 代码评审一次通过率

## 4. 原理是什么（工程视角）

其底层原理可以概括为：

1. 降低上下文熵
- 把模糊任务转成结构化输入，减少模型输出漂移。

2. 约束输出边界
- 通过模板里的“必须项/禁止项”限制无关改动，提升可控性。

3. 放大复利效应
- 一次模板优化可在后续所有任务复用，边际成本持续下降。

4. 人机分工清晰
- AI 负责初稿与批量化执行，人类负责取舍、验收和关键决策。

## 5. 适用场景

- 有持续迭代需求的个人开发者/小团队
- 希望把“提示词能力”升级为“可交付流程”的工程团队
- 需要新成员快速对齐 AI 编程规范的项目

## 6. 风险与注意事项

1. 模板依赖风险
- 模板质量会直接决定输出质量，必须版本化维护。

2. 过度自动化风险
- 若省略人工验收，容易把错误规模化扩散。

3. 迁移风险
- 不同模型/版本行为差异明显，模板应做兼容性回归。
