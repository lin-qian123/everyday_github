# hunk（modem-dev/hunk）

- GitHub: <https://github.com/modem-dev/hunk>
- 核心定位：面向 AI 代码代理时代的“审查优先”终端 diff 查看器，用结构化 UI 取代纯文本补丁阅读。

## 1. 它解决什么问题

当团队大量使用 Codex/Claude Code 等代理写代码后，问题从“怎么改”变成“怎么高效审”。传统 `git diff` 在多文件、长补丁、并行改动下认知负担较高。

`hunk` 的价值在于把“补丁审查”抽象成独立工作流：
- 多文件连续审查；
- 侧边栏导航和布局切换；
- 适配 agent 改动密集场景。

## 2. 快速使用（实践向）

```bash
git clone https://github.com/modem-dev/hunk.git
cd hunk
# 按官方 README 安装并运行（项目迭代快，建议以仓库最新说明为准）
```

在团队中的建议落地方式：
- PR 前固定过一遍 `hunk` 审查流程，先看风险文件再看样式文件；
- 对自动生成或批量改动设独立审查路径，避免混在核心逻辑补丁里。

## 3. 原理拆解（简化）

- 终端 TUI：把 diff 呈现从线性文本切换到可导航视图；
- 多文件上下文：强调跨文件关联而非单文件局部优化；
- 评审优先设计：更贴近“AI 先产出、人工后把关”的新流程。

## 4. 为什么值得关注

- 直接切中 AI 编码工作流里的高频痛点：审查效率与遗漏风险；
- 对中大型补丁的理解速度提升明显；
- 适合和现有 Git 评审机制并行，不必重做主流程。

## 5. 风险与边界

- 只是“审查界面增强”，不替代测试与架构审查；
- 团队仍需建立一致的审查标准（安全、性能、回归、可维护性）。

## 6. 我的补充建议

- 给高风险模块建立“必检清单”（权限、资金、删除、并发）；
- 将 `hunk` 输出与 CI 报告绑定，减少“看了补丁但没看测试”的断层。
